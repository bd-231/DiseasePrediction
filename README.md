# DiseaseReco

A full-stack disease diagnosis and medication recommendation web application with doctor review and validation. This repository contains the backend API, frontend UI, sample datasets, and Jupyter notebooks used to prepare data and train models.

## Quick Summary

- Backend: FastAPI + SQLAlchemy
- Frontend: React + Vite + Tailwind
- Notebooks: data prep, EDA, model training located in `notebooks/`
- Data: raw CSVs in `data/raw/` (tracked), processed outputs generated by notebooks in `data/processed/` (ignored)
- Models: generated into `models/` by the notebooks (recommended to keep them out of Git; use Git LFS for large artifacts)

## Repository Layout

- `backend/` — FastAPI application, routers, services
- `frontend/` — React + Vite UI
- `notebooks/` — Jupyter notebooks (run in order to reproduce processed data and models)
- `data/raw/` — raw CSV dataset files (tracked)
- `data/processed/` — generated/processed files (ignored by git)
- `models/` — model artifacts produced by notebooks (should be managed with Git LFS or stored externally)

## Prerequisites

- Python 3.11
- Node.js 18+ and npm (or pnpm)
- Git
- (Optional but recommended) Git LFS for large model files

## Setup (development)

From the project root:

1. Create a Python virtual environment and activate it:

```bash
python3 -m venv .venv
source .venv/bin/activate
```

2. Install Python dependencies (project-wide):

```bash
pip install -r requirements.txt
# Jupyter/nbconvert are required to run notebooks
pip install jupyter nbconvert papermill
```

3. Install frontend dependencies (optional for backend-only work):

```bash
cd frontend
npm install
cd ..
```


Note: If you cannot use Git LFS, download the required model artifacts externally and place them into `models/`.

## Reproducing data and models (Notebooks)

All generated data in `data/processed/` and models in `models/` are produced by the notebooks in `notebooks/`. This repository tracks only `data/raw/` (original CSVs). To regenerate processed datasets and models, run the notebooks in order.

Recommended notebook order:

1. `notebooks/01_data_exploration.ipynb`
2. `notebooks/02_data_preprocessing.ipynb`
3. `notebooks/03_eda.ipynb` (optional)
4. `notebooks/04_disease_prediction_model_v2.ipynb`
5. `notebooks/05_medicine_recommendation_model_fixed.ipynb`

Run a single notebook in-place (from repo root):

```bash
jupyter nbconvert --to notebook --execute --inplace --ExecutePreprocessor.timeout=0 notebooks/01_data_exploration.ipynb
```

Run all notebooks in-order with a simple loop (Linux/macOS):

```bash
for nb in \
  notebooks/01_data_exploration.ipynb \
  notebooks/02_data_preprocessing.ipynb \
  notebooks/03_eda.ipynb \
  notebooks/04_disease_prediction_model_v2.ipynb \
  notebooks/05_medicine_recommendation_model_fixed.ipynb; do
  jupyter nbconvert --to notebook --execute --inplace --ExecutePreprocessor.timeout=0 "$nb"
done
```

Alternatives:

- Use `papermill` to parameterize notebooks and run them programmatically.
- If a notebook fails due to missing packages, install the package into the active venv and rerun.

Output:

- Processed CSVs will be written to `data/processed/` (created by notebooks).
- Trained models will be written to `models/`.

Important: `data/processed/` and `models/` are ignored by git to avoid checking generated artifacts into source control. Keep raw data in `data/raw/` only.

## Run Backend (development)

From the project root:

```bash
source .venv/bin/activate
cd backend
uvicorn app.main:app --reload --port 8000
```

By default the app will create `backend/healthcare.db` and seed a default admin user if none exists. To run the seeder manually:

```bash
python -c "from app.database import create_tables; from app.seed import seed_admin; create_tables(); from app.database import SessionLocal; db=SessionLocal(); seed_admin(db); db.close()"
```

API docs: `http://localhost:8000/docs`

## Run Frontend (development)

From the project root:

```bash
cd frontend
npm run dev
```

Default dev URL: `http://localhost:5173` (Vite)


## Troubleshooting

- If a notebook raises FileNotFoundError: confirm you ran notebooks from the repository root and that `DATA` is set to `data/raw/` inside the notebook.
- If the backend cannot find models: ensure `models/` contains the expected `.pkl` files and that `MODEL_DIR` points there.
